# Expectum Governance Rule 1.0

## Purpose

This rule protects Expectum's foundation before any further audit, refactor,
optimization, prompt work, memory work, design work, or feature work.

Expectum's foundation documents are:

- Master Context 4.0
- Design Log 1.0
- System Audit 1.0

All future work must remain aligned with these foundation documents.

The authoritative repository location and version-control process for the
foundation documents are not currently visible in the repository.

**Needs decision:** Where the authoritative copies of the foundation documents
are kept, who may update them, and how an approved change is versioned.

## Foundation Takes Precedence

Codex, ChatGPT, or any other development tool must not change, reinterpret, or
weaken the following without explicit human approval:

- Expectum identity
- Aim essence
- Memory purpose
- Kaja meaning
- the central role of Kohtumine
- the Kooskõla concept
- the symbol system
- Design Log reasoning
- Master Context principles

This protection applies equally to documentation and implementation work.
Technical convenience, current behavior, optimization, or consistency with an
existing implementation is not sufficient authority to alter the foundation.

If there is conflict between current implementation and foundation documents,
the system should move closer to the foundation documents, not the foundation
documents closer to the current implementation.

## Role of Audits

Audits may:

- identify conflicts between implementation and the foundation;
- document uncertainty or divergence;
- recommend the smallest safe steps that bring implementation closer to the
  foundation;
- identify a question that requires human decision.

Audits must not:

- revise foundation principles to match current implementation;
- present an inferred product principle as an established foundation principle;
- weaken the foundation in the name of simplification, consistency, or
  optimization;
- treat repeated implementation behavior as evidence that the foundation has
  changed.

When the available foundation material is insufficient to resolve a question,
the result must be marked **Needs decision** rather than answered by inference.

## Implication for Prompt Alignment

Aim essence remains constant. Aim expression may vary by meeting type.

Variation is allowed only when it preserves the foundation. Prompt alignment
must verify alignment with Master Context 4.0, Design Log 1.0, and System Audit
1.0 before changing wording or behavior.

Prompt work must not flatten all Aim expressions into one voice. It must not
remove meaningful differences between:

- ordinary meeting;
- thought meeting;
- avardamine;
- journey noticing;
- direction noticing;
- collective echo.

Those differences may be changed only when they contradict the foundation or
when explicit human approval establishes a new direction.

If the relevant foundation wording cannot be inspected, prompt alignment may
document the issue and recommend wording, but the unresolved interpretation
must be marked **Needs decision** before implementation.

## Human Approval Required

Explicit human approval is always required before:

- changing Master Context;
- changing Design Log;
- changing System Audit;
- changing the meaning of Aim;
- changing the meaning of Kaja;
- changing the meaning of Memory;
- changing the role of Kohtumine;
- changing the symbol system;
- changing whether Expectum is a space rather than a tool.

Approval must identify the intended change. A general request to improve,
refactor, simplify, align, optimize, or modernize Expectum does not authorize a
foundation change.

**Needs decision:** What form of human approval is authoritative, where that
approval is recorded, and whether foundation-changing work requires a separate
governance review.

## Working Rule

Before beginning any audit, refactor, optimization, prompt change, memory
change, design change, or feature change:

1. Identify which foundation principles the work may affect.
2. Verify the proposal against the available foundation documents.
3. Separate implementation alignment from foundation change.
4. Mark unresolved foundation questions **Needs decision**.
5. Stop and request explicit human approval if the proposal would change,
   reinterpret, or weaken a protected principle.

This rule does not prevent Expectum from changing. It ensures that change is
deliberate, traceable, and human-approved when it touches the foundation.
